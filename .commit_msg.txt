fix(acp): demote usage_update + session_info_update to best-effort so a stalled client cannot wedge prompt()

ACP ``HermesACPAgent.prompt()`` is the single entry point for every turn
the agent takes inside an ACP-capable IDE (JetBrains IDEA 2026.1.x, Zed,
VS Code). After a successful run_conversation the prompt() coroutine
emits two non-critical telemetry notifications before returning
PromptResponse:

  * usage_update        - context-window indicator
  * session_info_update - title refresh, post-rotation provenance

Both call ``await self._conn.session_update(...)`` directly with no
timeout. If the connected ACP client is not draining the notifications
promptly (a long-standing IDE bug at session transitions, multi-monitor
tool windows, backgrounded UI), the client stalls the awaited coroutine
and ``prompt()`` never gets to ``return PromptResponse(...)``.

The user-visible result: the IDE shows "Waiting…" indefinitely even
though Hermes actually finished the turn and produced a final response.
The only recovery has been to restart the IDE — there's no in-product
timeout/heartbeat fallback. Reported in #39245 (general ACP client
stalls) and the more specific #51290 (the hang persists after the first
turn whenever the created-session background review fires, because
state.is_running is set/cleared in unusual paths).

This PR demotes both updates to deterministic best-effort helpers that
share a single 3-second timeout (#51290 calls out usage + session info
together as the cruise-control layer around prompt()):

  * ``_send_usage_update_best_effort(state)``
  * ``_send_session_info_update_best_effort(session_id, *, current_hermes_session_id=None, previous_hermes_session_id=None)``

The internals are ``asyncio.wait_for(self._send_*(...), timeout=3.0)``
plus a ``logger.warning`` on TimeoutError and a bare ``except Exception``
on the outer wrap so callers can never observe a hang. Errors log inside
the original ``_send_*`` methods (which we deliberately don't touch —
this keeps the existing telemetry code paths and tests valid).

Three call sites in prompt() switch to the best-effort helpers:

  1. Slash-command path (after ``update_agent_message_text``).
  2. Post-rotation provenance update path.
  3. The final pre-return ``await self._send_usage_update(state)``.

Tests in ``tests/acp/test_prompt_best_effort_updates.py`` cover the
helpers directly with a stub ``session_update`` that hangs for 60 s —
the assertions confirm the helpers complete inside ~3 s instead of
waiting the full 60 s. Existing ``tests/acp/test_server.py`` (80
tests) all still pass; the production path is unchanged outside the
three call-sites listed above.

Cherry-picks + extends the still-open
https://github.com/NousResearch/hermes-agent/pull/39378
(``fix(acp): add timeout to usage_update before returning PromptResponse``)
which only wrapped the slash-command call-site; this PR also wraps the
final pre-return call-site (line ~1736) and adds session_info_update to
the same wrap. ``_send_usage_update`` itself and its internal behavior
remain backwards-compatible (``tests/acp/test_server.py`` exercises it
unmodified via the unchanged ``_send_usage_update`` path under
``_send_usage_update_best_effort``).

Fixes #39245, fixes #51290.
